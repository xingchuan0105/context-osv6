# Desktop 客户端 — 支持与 SLA

> 依据 ADR 0006 §9：桌面与云端 **独立版本线、独立 SLA**。  
> 状态：**Accepted draft** — 响应数字为产品运营初值，可经 changelog 修订，修订需同步销售/支持话术。

## 产品边界

| 项目 | 桌面 | 云端 SaaS |
|------|------|-----------|
| 主要交付物 | 可安装客户端（Tauri 等） | 托管 API + Web |
| 版本线 | 独立版本号与 changelog（见 `VERSIONING.md`） | 独立版本号与 changelog |
| LLM 用量 | **本地 / BYOK 不上报、不计入云配额** | User rolling 配额 |
| 连云功能 | 可选；走云时适用 **云端 SLA** | 本 SLA 不适用 |

## 版本线

- 桌面版本号与云 API **解耦**（例如 Desktop 1.x 可对应最低云 API `v1`）。
- 发布说明分列：
  - 云端：`docs/release/cloud/`（待启用时创建）
  - 桌面：`docs/release/desktop/`（待启用时创建）
- 破坏性变更：桌面 changelog 必须注明「最低兼容云 API 版本」（若该版本含连云能力）。
- 兼容矩阵草案见 `VERSIONING.md`。

## 支持范围

### 我们负责

- 安装包完整性、许可激活/校验缺陷  
- 桌面 UI/本地功能崩溃、明确的产品缺陷  
- 文档中承诺的连云能力在 **云端可用** 时的客户端联调问题  

### 默认不在本 SLA 内（用户侧）

- BYOK API Key 错误、厂商限流、余额不足  
- 用户本机模型进程/GPU/环境配置  
- 用户修改系统导致的客户端无法启动（可 best-effort 协助）  
- **本机进程「可用性 99.x%」** — 不承诺  

## 响应目标

工作日按 **UTC+8 工作时间 09:00–18:00** 起算（法定节假日顺延）。严重级别由支持值班与工程共同判定。

| 严重级别 | 含义 | 目标首次响应 | 目标处理（workaround 或修复路径） |
|----------|------|--------------|----------------------------------|
| S1 | 许可体系大面积失效 / 已确认安全漏洞 | **4 工作小时** | **1 工作日** 内给出缓解或修复计划；安全类按负责任披露流程 |
| S2 | 主路径不可用（安装后无法完成核心任务） | **1 工作日** | **3 工作日** 内给出 workaround 或修复版本计划 |
| S3 | 功能缺陷有绕行 | **2 工作日** | 纳入最近一个合理发布窗口 |
| S4 | 体验/文案 | **5 工作日** | 产品排期，不保证版本 |

**说明**

- 「目标」非违约金式合同承诺，除非客户合同另有书面约定。  
- S1/S2 若根因在云端，桌面支持负责衔接；**可用性与数据**以云端 SLA 为准。  

## 与云端 SLA 的关系

- 用户登录云并调用云 API 时：**服务端可用性与数据** 以 **云端 SLA** 为准。  
- 桌面仅保证：在云端正常时，客户端能按文档完成认证与调用编排。  
- 本地 LLM / BYOK 故障：**默认用户侧**；产品缺陷除外。  

## 升级政策（摘要）

- 主版本（x.0）可含破坏性变更；须在 release notes 标明。  
- 补丁（x.y.z）用于安全与关键缺陷修复；S1 可发紧急补丁。  
- 旧主版本支持窗口：至少覆盖 **上一个主版本 6 个月**（可随产品调整）。
