## Subagents 编排建议

此任务适合使用 subagents，但只适合“协调式、分阶段”的编排，不适合多个 worker 同时自由写代码。原因是剩余 GAP 大多跨 `app` agent runtime、GraphFlow tasks、search crate、common contract 和 rag-core，共享边界多，盲目并行会产生冲突或重新引入双路径。

建议编排方式：

- 用户指定的 subagent 配置覆盖默认模型映射：只读 explorer 使用 `GPT5.3codexspark`，写代码 worker 使用 `GPT5.4mini-fast`，reviewer 使用 `GPT5.5fast`，所有 subagent 的推理等级均为 `xhigh`。
- 先启动只读 explorer：分别确认 Chat fallback 残留、Search provider/config 调用链、RAG execute topology 与测试覆盖，输出文件清单和风险点，不写文件。
- Worker A：只处理 Chat/general 收尾，写入范围限定为 `crates/app/src/chat/*`、相关 app/transport tests；完成后跑 Chat focused tests。
- Worker B：在 Worker A 合并后处理 WebSearch，写入范围限定为 `crates/search/*`、`crates/app/src/agents/web_search_agent.rs`、search config/bootstrap、`.env.example` 和 search tests；不同时改 RAG。
- Worker C：先单独处理 RAG common contract，写入范围限定为 `crates/common/src/rag_execute.rs` 和 `crates/common/tests/rag_execute_contract.rs`；通过 common tests 后再继续。
- Worker D：在 Worker C 后处理 RAG app/rag-core topology，写入范围限定为 `graphflow_tasks_rag.rs`、`lib_impl/rag_execute.rs`、`rag-core/src/runtime/*`、RAG focused tests；不同时改 Search。
- Reviewer agent：每个 worker 完成后做 spec compliance review，重点检查是否保留 legacy 主路径、是否出现 fake streaming、是否扩大 doc_scope、是否泄漏 API key/debug payload。
- QA agent：最后只做验证与汇总，跑 focused tests、前端 typecheck/tests、diff/secrets scan，记录未跑项。

不建议交给 subagents 并行写的部分：

- `UnifiedAgentService`、`AgentEvent`、`SseSink`、`chat_streaming.rs` 等事件/streaming 核心边界。
- `service_modes.rs` 与 `graphflow_tasks_*` 的同一阶段修改。
- RAG contract 和 rag-core topology 同时改，除非明确拆成先 contract 后 runtime 的顺序。
- MainAgent cleanup，不应提前并行执行，只能等 Chat/Search/RAG gates 都通过后再做。
